# Render Deployment

## Recommended layout

Deploy the backend as three Render resources:

- `cargopilot-postgres` as Render Postgres
- `cargopilot-api` as a Docker web service
- `cargopilot-label-worker` as a Docker background worker

The repo now includes `render.yaml`, so the easiest path is a Render Blueprint deploy.

## First deploy

1. Push the backend repo to GitHub.
2. In Render, click `New` -> `Blueprint`.
3. Select the backend repository.
4. Render will detect `render.yaml` and propose:
   - one Postgres database
   - one web service
   - one worker
5. Fill the prompted secrets:
   - `CLIENT_URL`
   - `AWS_S3_BUCKET`
   - `AWS_ACCESS_KEY_ID`
   - `AWS_SECRET_ACCESS_KEY`
6. Click `Apply`.

## Required values

The blueprint already wires these automatically:

- `DATABASE_URL` from Render Postgres
- `JWT_SECRET` generated by Render
- `NODE_ENV=production`
- `PORT=4000`
- `PAYMENTS_ENABLED=false`

You must provide these at creation time:

- `CLIENT_URL`
  - example: `https://app.your-domain.com`
- `AWS_S3_BUCKET`
- `AWS_ACCESS_KEY_ID`
- `AWS_SECRET_ACCESS_KEY`

## Optional payment values

Payments are disabled by default in the blueprint.

If you enable Stripe later, add these env vars to `cargopilot-api`:

- `STRIPE_SECRET_KEY`
- `STRIPE_WEBHOOK_SECRET`
- `PAYMENTS_ENABLED=true`

## After deploy

Run these checks:

1. Open `https://your-api-domain/api/health`
2. Log into the frontend against the new API
3. Create one order
4. Open order details
5. Confirm label generation works
6. Check worker logs for label processing errors

## Auto deploys

`render.yaml` currently uses:

- `autoDeployTrigger: commit`

That means every push to the connected branch will redeploy automatically.

If you later add GitHub Actions and want stricter CI/CD, change both services to:

```yaml
autoDeployTrigger: checksPass
```

Then let GitHub Actions run the checks before Render deploys.
